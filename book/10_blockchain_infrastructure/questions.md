**Q102.** What are the main consensus mechanisms and how do they differ?

Proof of Work (PoW) has miners compete to solve a SHA-256 puzzle — the winner adds the block and earns the reward. Security comes from computation cost; an attacker needs 51% of total hash power. It wastes enormous energy and has slow probabilistic finality. Proof of Stake (PoS) has validators lock up cryptocurrency as collateral; a validator is pseudorandomly selected to propose a block weighted by stake, others attest, and misbehavior is punished by slashing. BFT/Tendermint uses a known validator set that votes in rounds — a block is finalized when 2/3+ validators agree, giving instant deterministic finality. The key distinction is finality type: PoW and Ethereum PoS are probabilistic (a block buried under many blocks is practically irreversible but theoretically reversible), while BFT is deterministic (once 2/3 validators sign, it's final — no reorg possible).

**Q103.** What is slashing and why does it exist in Proof of Stake?

Slashing is the protocol-level destruction of a validator's staked funds as punishment for provably malicious behavior. The two main slashable offenses are double-voting (signing two different blocks at the same height — equivocation) and surround voting (attempting to contradict a previous finalization). Slashing creates a strong economic disincentive: attacking the network requires not just acquiring 33%+ of stake but also risking losing it all. Without slashing, PoS has a "nothing-at-stake" problem — validators could safely vote on multiple competing forks with no cost. Slashing ties the cost of attack directly to the value of the stake, making attacks economically irrational under normal conditions.

**Q104.** Walk through the full lifecycle of a transaction from user signing to finality.

The user constructs and signs a transaction locally with their private key, setting `maxFeePerGas` and `maxPriorityFeePerGas`. The signed transaction is broadcast to the network and lands in the public mempool — a shared pool of unconfirmed transactions visible to all nodes. Validators and block builders observe the mempool, select transactions (typically by fee priority), order them, and build a candidate block. The builder proposes the block to the validator network. Other validators attest or vote to confirm the block is valid. Once enough attestations are collected (2/3+ stake in Ethereum), the block is finalized and added to the canonical chain. Transaction receipts become available — containing gas used, status, emitted events, and logs. On Ethereum, a block is "safe" after ~12 seconds (1 slot) and "finalized" after ~12.8 minutes (2 epochs).

**Q105.** What is a nonce and why does nonce management matter for backend systems?

A nonce is a sequential counter attached to every transaction sent from an Ethereum account. Each transaction must have a nonce equal to the current account nonce — if it's too low the transaction is rejected as a replay, if it's too high it queues and won't execute until the gap is filled. For backend systems sending many transactions concurrently (e.g., relayers, bots, settlement engines), nonce management is critical: two goroutines accidentally using the same nonce causes one transaction to silently overwrite the other. The correct pattern is a mutex-protected nonce manager that assigns nonces atomically, increments the local counter immediately on submission (without waiting for RPC confirmation), and handles failures by detecting whether the nonce was already consumed or needs retry.

**Q106.** What is the difference between the UTXO model and the account model?

In the UTXO (Unspent Transaction Output) model used by Bitcoin, there are no accounts — only individual coins (UTXOs) that can be consumed once. A transaction takes UTXOs as inputs and creates new UTXOs as outputs; your balance is the sum of UTXOs you control. Each transaction looks different, which provides privacy benefits. However, smart contracts are nearly impossible to implement cleanly in this model. In the account model used by Ethereum and Starknet, there are explicit accounts with a balance and nonce stored in a global state trie. A transaction decrements the sender's balance and increments the receiver's. It's intuitive, natural for smart contracts, and easy to reason about — but full history is easily traceable and the sequential nonce makes concurrent transaction sending difficult without a nonce manager.

**Q107.** What is Account Abstraction and what problems does it solve?

Traditional Ethereum only allows Externally Owned Accounts (EOAs) — secp256k1 key pairs — to initiate transactions. Account Abstraction (AA) makes every account a smart contract that defines its own validation logic, enabling: social recovery (trusted parties can restore access if a key is lost), multisig by default (require M-of-N signers for every transaction), gas sponsorship via paymasters (a third party pays gas, enabling gasless UX), session keys (temporary limited-permission keys for dApps or games), batch transactions (many operations in one call), and custom signature schemes (passkeys, WebAuthn, hardware keys). ERC-4337 implements AA on Ethereum without protocol changes — users submit `UserOperation` objects to a `Bundler` which batches them and calls the `EntryPoint` contract. On Starknet, AA is native: every account is a smart contract from day one.

**Q108.** What is the difference between multisig and MPC wallets?

Multisig (multi-signature) uses a smart contract requiring M-of-N signatures before executing a transaction. The logic is enforced on-chain (Gnosis Safe is the standard) — it's transparent, auditable, and battle-tested. The downsides: all signers are visible on-chain, each additional signature increases gas cost, and it requires coordination between signers. MPC (Multi-Party Computation) splits the private key into cryptographic shares using protocols like threshold ECDSA — shares are distributed to M-of-N parties who jointly produce a signature without ever assembling the full key or revealing shares to each other. The resulting signature looks like a normal single-key signature on-chain, providing privacy and gas efficiency. The downside is cryptographic complexity and harder auditability. Multisig is preferred for DAO treasuries and on-chain governance where transparency matters; MPC is preferred for institutional custody (Fireblocks, Anchorage) and exchange hot wallets where privacy and gas efficiency matter.

**Q109.** Explain the CAP theorem and how it applies to blockchains.

The CAP theorem states that a distributed system can guarantee at most 2 of 3 properties: Consistency (every read returns the most recent write), Availability (every request gets a response, though not necessarily the latest data), and Partition tolerance (the system continues working even if the network splits). Since network partitions are inevitable in real distributed systems, the practical choice is between CP (consistent but may become unavailable during a partition) and AP (always available but may return stale data). Most blockchains are CP: during a network partition, honest nodes will not finalize conflicting blocks (preserving consistency) but may halt block production (becoming temporarily unavailable). This is intentional — it's safer to stop than to allow divergent state that would need to be reconciled.

**Q110.** What is the difference between a full node, archive node, and light node?

A full node downloads and validates every block and transaction from genesis, maintaining the current chain state (balances, contract storage) needed to validate new blocks. An archive node is a full node that additionally stores all historical state snapshots at every block — required for queries like `eth_getStorageAt` at a past block number. Archive nodes are massive (Ethereum: 10TB+) and used by block explorers, analytics services, and protocols needing historical state. A light node downloads only block headers and trusts full nodes to answer state queries — it verifies inclusion proofs but cannot validate state independently. Light nodes are used in mobile wallets and environments where storage is limited, at the cost of trusting the RPC provider.

**Q111.** What are the main security risks in blockchain backend systems and how do you mitigate them?

The main risks are: (1) **Private key exposure** — never store keys in environment variables in production; use HSM or cloud KMS (AWS KMS, GCP KMS) with a dedicated signer service. (2) **Nonce race conditions** — two concurrent threads sending transactions can use the same nonce, causing one to be dropped; use a mutex-protected nonce manager. (3) **Signature replay attacks** — a signed message reused on another chain or with a different nonce; always include `chainId` and `nonce` in signed messages. (4) **Stale on-chain state** — never trust cached or off-chain data for critical decisions; validate on-chain state with `eth_call` before sending a transaction. (5) **Trusting return values** — some old ERC-20 contracts return `false` on failure instead of reverting; always check return values. (6) **User-supplied contract addresses** — an attacker can pass a malicious contract address; whitelist known contracts and never call arbitrary addresses with user-controlled data.
